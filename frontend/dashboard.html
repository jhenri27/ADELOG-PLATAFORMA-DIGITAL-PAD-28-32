Reglas Estrictas de Integridad y Cumplimiento

Esta sección enumera los controles automáticos y de auditoría integrados bajo la normativa PLAD para blindar el padrón electoral.

Regla Anti-Duplicidad de Cédulas

El motor de base de datos impone una restricción única (`UNIQUE KEY`) sobre el campo de cédula. Esto imposibilita que el mismo ciudadano sea inscrito más de una vez en el padrón 2028, evitando la inflación artificial de metas.

Validación Estructural de Identidad (Algoritmo de Luhn)

El sistema valida matemáticamente el dígito verificador de cada cédula ingresada aplicando el Algoritmo de Luhn dominicano. Bloquea inscripciones con numeraciones inválidas antes de enviar peticiones al servidor.

Consulta de Padrón y Filtro de Existencia (DGII API)

La plataforma integra un handshake cURL contra la API del validador de contribuyentes (DGII) para auditar la existencia real del ciudadano, asegurando que las identidades inscritas correspondan a personas físicas reales.

Validación del Colegio Electoral (COEL Restrictivo)

Cada colegio es comparado contra la estructura JCE 2024. Si el colegio ingresado no coincide con ninguna Región/Zona/Circunscripción oficial, el sistema bloquea el registro directo y ofrece canalizarlo en estado `Pendiente de Regularización` para fidelización.

Centro de Respaldos y Push Update a GitHub

Administración de copias de seguridad en caliente hacia unidades externas y sincronización automatizada con el repositorio ADELOG en GitHub.

1. Copias de Seguridad (Backup)

Genera una copia en caliente de la base de datos (estructura vacía de electores para proteger la privacidad) y guarda los scripts del sistema en:
F:\\ADELOG\\PLATAFORMA DIGITAL-PAD-28-32\\backups

2. Push Update a GitHub

Sincroniza y sube de forma instantánea todos los cambios del código fuente local y las copias de seguridad generadas al repositorio remoto de GitHub **ADELOG**.
